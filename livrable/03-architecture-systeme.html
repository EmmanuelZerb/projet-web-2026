1. Vue d'ensemble de l'architecture

ECE In est une application web monolithique PHP suivant une architecture 3 tiers : couche presentation (client), couche logique (serveur PHP) et couche donnees (MySQL).

4. Flux de donnees

4.1 Flux d'une requete standard (page PHP)

sequenceDiagram
    participant N as Navigateur
    participant P as Page PHP
    participant C as config.php
    participant D as database.php
    participant M as MySQL

    N->>P: GET /index.php
    P->>C: require config.php
    C->>C: session_start(), constantes, helpers
    C->>D: getDB()
    D->>M: Connexion PDO (MySQL)
    M-->>D: Connexion etablie
    P->>M: SELECT publications, evenements, suggestions
    M-->>P: Resultats (PDOStatement)
    P->>P: include header.php, navbar.php
    P->>P: Genere le HTML avec les donnees
    P->>P: include footer.php
    P-->>N: Reponse HTML complete

4.2 Flux AJAX (API endpoint)

sequenceDiagram
    participant N as Navigateur (JS)
    participant A as api/posts.php
    participant C as config.php
    participant M as MySQL

    N->>A: POST /api/posts.php (action=publier, FormData)
    A->>C: require config.php
    C->>C: Verification session (requireConnexion)
    A->>M: INSERT INTO publications (...)
    M-->>A: OK (lastInsertId)
    A->>M: INSERT INTO notifications (...)
    M-->>A: OK
    A-->>N: JSON { success: true, post_id: 42 }
    N->>N: Mise a jour du DOM (ajout du post)

4.3 Flux d'authentification

sequenceDiagram
    participant N as Navigateur
    participant L as login.php
    participant C as config.php
    participant M as MySQL

    N->>L: POST /login.php (pseudo, mot_de_passe)
    L->>C: require config.php
    L->>M: SELECT * FROM utilisateurs WHERE pseudo = ?
    M-->>L: Utilisateur trouve
    L->>L: password_verify(mdp, hash_bcrypt)
    alt Mot de passe correct
        L->>L: $_SESSION[utilisateur_id] = id
        L->>M: UPDATE derniere_connexion = NOW()
        L-->>N: Redirect 302 → index.php
    else Mot de passe incorrect
        L-->>N: HTML avec message d'erreur
    end

4.4 Flux messagerie (polling)

sequenceDiagram
    participant N as Navigateur (JS)
    participant A as api/messages.php
    participant M as MySQL

    loop Toutes les 3 secondes
        N->>A: GET /api/messages.php?action=poll&conv=5&last=42
        A->>M: SELECT * FROM messages WHERE conversation_id=5 AND id > 42
        M-->>A: Nouveaux messages (ou vide)
        A-->>N: JSON [ {id:43, contenu:"..."}, ... ]
        N->>N: Afficher les nouveaux messages dans le DOM
    end

5. Securite de l'architecture

MecanismeImplementationCouche
Authentification Sessions PHP securisees (httponly, samesite=Strict), bcrypt pour les mots de passe Serveur
Autorisation Fonctions requireConnexion() et requireAdmin() en debut de chaque page/API Serveur
Injection SQL Requetes preparees PDO avec placeholders (?) partout Base
XSS (Cross-Site Scripting) Fonction h() (htmlspecialchars) sur toutes les sorties utilisateur Serveur
Restriction inscription Emails limites aux domaines @edu.ece.fr et @ece.fr Serveur
Upload securise Validation MIME type + taille maximale (PHOTO_TYPES, MAX_PHOTO_SIZE) Serveur
Visibilite des posts 3 niveaux (public, amis, prive) verifies cote serveur dans les requetes SQL Base
