1. Description complete du projet

1.1 Qu'est-ce qu'ECE In ?

ECE In est un reseau social professionnel concu pour la communaute de l'ECE Paris. Inspire de LinkedIn, il permet aux etudiants, enseignants, alumni et personnel de l'ECE de :

  • Creer un profil professionnel avec formations, projets et competences
  • Publier du contenu : statuts, photos, videos, evenements, CV
  • Gerer un reseau d'amis professionnels avec demandes de connexion
  • Communiquer par messagerie privee en temps reel (+ appels audio/video)
  • Consulter et filtrer des offres d'emploi (stages, alternances, CDI...)
  • Generer un CV automatiquement a partir des donnees du profil (HTML, PDF, XML)
  • Recevoir des notifications en temps reel

2.2 Architecture 3 tiers

L'application suit une architecture classique en 3 couches :

  1. Client (Navigateur) : HTML/CSS/JS → envoie des requetes HTTP et AJAX
  2. Serveur (PHP) : traite les requetes, genere le HTML ou retourne du JSON
  3. Donnees (MySQL) : stocke toutes les donnees utilisateurs, posts, messages, etc.

2.3 Securite

  • Mots de passe : haches avec bcrypt via password_hash()
  • Injection SQL : requetes preparees PDO avec placeholders ?
  • XSS : fonction h() (htmlspecialchars) sur toutes les sorties
  • Sessions : httponly, samesite=Strict
  • Inscription : limitee aux emails @edu.ece.fr et @ece.fr
  • Uploads : validation MIME + taille max

2.4 Base de donnees (16 tables)

utilisateurs, formations, projets, publications, reactions, commentaires, connexions, conversations, conversation_participants, messages, notifications, emplois, evenements, albums, album_photos, appels

Point cle a retenir : Toutes les FK utilisent ON DELETE CASCADE. La suppression d'un utilisateur supprime automatiquement toutes ses donnees associees.

6. Questions du prof et reponses preparees

6.1 Questions techniques

Q: Pourquoi avoir choisi PHP plutot qu'un framework comme Laravel ou Node.js ?
Le sujet impose PHP et jQuery comme technologies. Nous avons choisi du PHP natif pour mieux comprendre les mecanismes sous-jacents (sessions, PDO, requetes preparees) plutot que de nous appuyer sur un framework qui masque cette complexite. Bootstrap 5 nous a fourni le framework CSS necessaire pour le responsive.
Q: Comment fonctionne votre systeme d'authentification ?
L'utilisateur entre son pseudo ou email + mot de passe. On recherche l'utilisateur en base, on verifie le mot de passe avec password_verify() (bcrypt). Si valide, on cree une session PHP avec les infos utilisateur ($_SESSION['utilisateur_id'], pseudo, role). Chaque page appelle requireConnexion() qui verifie la session. Les sessions sont configurees en httponly et samesite=Strict pour la securite.
Q: Comment gerez-vous les injections SQL ?
Toutes nos requetes utilisent PDO avec des requetes preparees et des placeholders ?. On ne concatene jamais de variables directement dans les requetes SQL. Par exemple : $stmt = $db->prepare("SELECT * FROM utilisateurs WHERE pseudo = ?"); $stmt->execute([$pseudo]);
Q: Comment fonctionne la messagerie en temps reel ?
Nous utilisons du polling AJAX : le navigateur interroge le serveur toutes les 3 secondes via api/messages.php en envoyant l'ID du dernier message recu. Le serveur retourne les nouveaux messages en JSON, et le JavaScript met a jour le DOM. Ce n'est pas du vrai temps reel comme les WebSockets, mais c'est fonctionnel et simple a implementer avec PHP.
Q: Pourquoi avoir choisi le polling plutot que les WebSockets ?
Les WebSockets necessitent un serveur persistant (Node.js, Ratchet en PHP), ce qui complique le deploiement et sort du cadre du projet. Le polling toutes les 3 secondes est un bon compromis entre reactivite et simplicite, et il fonctionne bien avec le serveur PHP built-in.
Q: Comment gerez-vous la visibilite des publications ?
Chaque publication a un champ visibilite (ENUM : public, amis, prive). Cote serveur, dans la requete SQL du fil d'actualite, on filtre : les posts publics sont visibles par tous, les posts "amis" sont visibles uniquement si l'auteur est dans le reseau de l'utilisateur connecte (jointure avec la table connexions, statut "accepte"), et les posts "prive" ne sont visibles que par l'auteur.
Q: Comment fonctionne la generation de CV ?
Le fichier cv/generer_cv.php recupere les donnees du profil (informations, formations, projets) depuis MySQL et genere le CV en 3 formats : HTML (affichage web dans un iframe), PDF (via l'impression du navigateur window.print()), et XML (export structure). On peut aussi importer un fichier XML via api/import_cv.php pour remplir automatiquement les formations et projets du profil.
Q: Expliquez votre modele entite-association.
Notre MCD contient 16 entites centrees autour de la table UTILISATEURS. Les relations principales : un utilisateur possede des formations et projets (1,N), publie des publications (1,N), a des connexions bidirectionnelles avec d'autres utilisateurs, participe a des conversations (N,M via table de jonction), et peut creer des albums qui regroupent des publications photo (N,M). Les reactions ont une contrainte UNIQUE (publication_id, utilisateur_id) pour empecher les doublons.
Q: Qu'est-ce que ON DELETE CASCADE et pourquoi l'utiliser ?
ON DELETE CASCADE signifie que quand on supprime un enregistrement parent, tous les enregistrements enfants lies par la cle etrangere sont automatiquement supprimes. Par exemple, si on supprime un utilisateur, toutes ses publications, reactions, commentaires, messages et formations sont supprimes automatiquement. Cela garantit l'integrite referentielle sans avoir a gerer manuellement les suppressions en cascade dans le code PHP.

6.2 Questions sur le projet

Q: Comment avez-vous reparti le travail ?
Emmanuel s'est charge du front-end (integration Bootstrap, pages PHP, messagerie, deploiement Railway) et Nikola de la conception back-end (MCD, schema SQL, authentification, panel admin, CV, specs fonctionnelles). Nous avons utilise Git pour la collaboration et nous nous sommes coordonnes regulierement.
Q: Quelle a ete la difficulte principale ?
Le deploiement sur Railway a ete le plus complexe. Le systeme de fichiers est ephemere (les uploads disparaissent au redemarrage), ce qui nous a obliges a stocker les photos en LONGBLOB directement dans la base de donnees. De plus, la gestion des variables d'environnement MySQL (MYSQL_URL vs variables separees) a necessite plusieurs iterations de debug.
Q: Qu'est-ce que vous feriez differemment si vous recommenciez ?
On utiliserait WebSockets (avec Node.js ou Ratchet) pour la messagerie temps reel, un systeme de stockage cloud (S3) pour les fichiers, et on ajouterait des tests automatises. On pourrait aussi utiliser un ORM pour simplifier les requetes SQL et un systeme de cache (Redis) pour les performances.
Q: Comment avez-vous utilise l'IA dans ce projet ?
Nous avons utilise Cursor IDE avec le modele Claude pour le brainstorming, le debogage, et des suggestions de fragments de code (structures CSS, logique de formulaires, requetes SQL). Environ 35% du code a ete influence par l'IA, mais tout a ete relu, compris et adapte par l'equipe. Les choix d'architecture et la logique metier sont entierement les notres.
Q: Quelles sont les limites de votre application ?
Le polling n'est pas du vrai temps reel. Les appels audio/video sont enregistres en base mais ne font pas de veritable communication WebRTC. Il n'y a pas de pagination sur le fil d'actualite (tous les posts sont charges). La recherche est basique (LIKE SQL, pas de full-text). Et le stockage en LONGBLOB n'est pas optimal pour de gros volumes de fichiers.
Q: Comment fonctionne le systeme de notifications ?
Les notifications sont creees automatiquement par le code PHP lors de certaines actions : demande d'ami, acceptation, reaction, commentaire, partage, nouvel evenement, nouvelle offre. Elles sont stockees dans la table notifications avec un type, un message, un lien et un boolean "lue". Le badge dans la navbar affiche le nombre de notifications non lues, et la page notifications permet de les consulter et de les marquer comme lues.

6.3 Questions pieges

Q: Montrez-moi dans le code ou vous faites X (ex: la verification bcrypt).
Sois pret a ouvrir le code ! La verification bcrypt est dans login.php avec password_verify($mdp, $user['mot_de_passe']). Les requetes preparees sont dans tous les fichiers PHP ($db->prepare("...")->execute([...])). Le filtre de visibilite est dans index.php dans la requete SQL du fil d'actualite.
Q: Pourquoi tous les commits sont d'un seul auteur ?
Nikola a travaille principalement sur la conception (MCD, specs, documentation) et sur des taches qui ne generent pas de commits (tests fonctionnels, corrections). Emmanuel a fait la majorite des commits car il gerait l'integration du code et le deploiement. Nous aurions du mieux repartir les commits.
Q: Ce code a-t-il ete genere par l'IA ?
L'IA a ete utilisee comme outil d'assistance (brainstorming, debugging, suggestions de fragments) mais pas comme generateur complet. Nous pouvons expliquer chaque partie du code car nous l'avons relu, compris et adapte. Le journal d'assistance IA dans le PPT detaille exactement ou et comment l'IA a ete utilisee.
